# 구현 계획 초안: 나는 어떤 디저트형 연애일까?

## 현재 상태
- 저장소에는 아직 실제 Next.js 앱 코드가 없다.
- 현재 단계의 목표는 상세 기획서를 바탕으로 MVP 범위와 구현 구조를 고정하는 것이다.

## 제품 고정 제약
- Next.js 사용
- TypeScript strict 사용
- 총 24문항 유지
- 총 16유형 유지
- 질문 플로우 중 광고 금지
- 결과 및 콘텐츠 페이지에서만 광고 허용
- 모바일 우선 UX 유지
- 완료 기준으로 `lint`, `typecheck`, `tests`, `build` 통과 필요

## 권장 기술 방향
- Next.js App Router 기반 구조 사용
- 서버 컴포넌트 우선 설계
- 결과별 메타데이터 구성이 쉬운 라우팅 구조 채택
- 스코어링 로직, 결과 프로필, 공유 데이터, 광고 슬롯을 독립 모듈로 분리
- 모바일 퍼스트 스타일 시스템과 재사용 가능한 UI 컴포넌트 구조 채택

## MVP 정보 구조
- 랜딩 페이지
- 테스트 소개 및 시작 CTA
- 24문항 질문 플로우
- 16유형 결과 페이지
- 공유 영역
- 궁합 확장 CTA
- 결과 및 정보성 페이지 광고 슬롯

## 제안 모듈 분리
- 질문 데이터 모듈: 질문 문구, 선택지, 축, 점수 매핑
- 결과 유형 데이터 모듈: 16유형 프로필, 요약, 장점, 주의 포인트, 궁합, 데이트 무드
- 스코어링 모듈: 축별 점수 누적과 유형 판정
- 공유 데이터 모듈: 공유 문구, OG 메타, 결과 카드 데이터
- 랜딩 UI 컴포넌트
- 질문 카드 및 진행률 UI 컴포넌트
- 결과 배너 및 상세 섹션 UI 컴포넌트
- 광고 슬롯 컴포넌트: 결과 및 콘텐츠 페이지 전용

## 라우팅 방향
- `/` 랜딩 페이지
- `/test` 질문 플로우
- `/result/[type]` 유형별 결과 페이지
- `/compatibility` 또는 결과 하단 CTA 기반 확장 영역
- `/content/...` 유형 설명 및 정보성 콘텐츠 확장 페이지

## 데이터 및 로직 원칙
- 질문 데이터는 JSON 또는 타입이 명확한 정적 데이터로 관리
- 각 선택지는 성향 축 점수에 매핑
- 결과는 결정론적으로 16유형 중 하나로 산출
- 스코어링 로직은 UI와 완전히 분리
- 결과 데이터와 공유 데이터는 재사용 가능하도록 분리

## UX 및 디자인 구현 방향
- 디저트샵 포스터 감성의 비주얼 톤 유지
- 귀엽지만 과하게 유아틱하지 않은 2030 타깃 스타일
- 질문은 1문항 1화면 또는 카드 전환형으로 집중도 유지
- 결과 첫 화면에서 유형명, 한 줄 요약, 대표 비주얼, 공유 CTA를 우선 노출
- 광고는 결과 해석을 읽은 뒤 자연스럽게 이어지도록 배치

## 검증 전략
- 스코어링과 유형 판정 순수 함수 테스트
- 24문항과 16유형 데이터 무결성 검증
- 질문 플로우에서 광고가 노출되지 않는지 검증
- 결과 페이지 메타데이터 및 공유 요소 검증
- 모바일 뷰포트 기준 UI 가독성 점검
- 빌드 및 타입 검사 자동화

## 보류 결정
- 16개 유형의 최종 카피와 설명문
- 카카오톡 공유 구현 방식
- 결과 카드 저장 방식
- 궁합 확장 기능의 1차 포함 여부
